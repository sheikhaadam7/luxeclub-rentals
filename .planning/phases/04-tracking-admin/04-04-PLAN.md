Read the existing `app/actions/admin.ts` (which by now includes Plan 03's additions). Add the following new Server Actions below the existing ones:

**KYC Actions:**

1. `getPendingKYC()` — Query profiles with `kyc_status IN ('submitted', 'pending')`. Select: `id, email` (from auth.users — or use admin client to query profiles joined with auth data). Since profiles may not have email directly, query `admin.auth.admin.listUsers()` or use admin client to query profiles and then separately look up emails. Simplest: use admin client to query profiles with kyc columns, and separately fetch user emails via `admin.auth.admin.listUsers()` to build a map. Return array of `{ userId: string, email: string, kycStatus: string, kycSessionId: string | null, createdAt: string }`. Order by `created_at ASC` (oldest first — FIFO queue).

   **Alternative simpler approach:** The profiles table has an `id` that matches `auth.users.id`. Use the admin Supabase client: query `profiles` for `kyc_status IN ('submitted', 'pending')`, then for each profile, get the user email via `admin.auth.admin.getUserById(profile.id)`. This is N+1 but the pending KYC queue will typically be small (< 50). If performance is a concern, batch with `admin.auth.admin.listUsers()` and filter.

2. `reviewKYC(userId, decision)` — Accept `userId: string` and `decision: 'verified' | 'rejected'`. Use optimistic locking: update profiles SET `kyc_status = decision`, `kyc_verified_at = decision === 'verified' ? new Date().toISOString() : null` WHERE `id = userId AND kyc_status IN ('submitted', 'pending')`. Check returned data length — if 0, return `{ error: 'Already processed or not in pending state' }`. Return `{ error: string | null }`.

**Payment Actions:**

3. `getPaymentBookings()` — Query all bookings where `payment_status IN ('pending_cash', 'unpaid')` OR all bookings with `payment_method IN ('cash', 'bank_transfer')`. This shows the admin bookings that need manual payment confirmation. Select: `id, vehicle_id, user_id, total_due, payment_status, payment_method, status, created_at`. Join `vehicles(name)`. Return array ordered by `created_at DESC`.

4. `confirmManualPayment(bookingId, method)` — Accept `bookingId: string` and `method: 'cash' | 'bank_transfer'`. Use optimistic locking: update bookings SET `payment_status = 'paid'`, `payment_method = method`, `updated_at = now()` WHERE `id = bookingId AND payment_status = 'pending_cash'`. Check data length — if 0, return `{ error: 'Booking not in pending cash state' }`. Return `{ error: string | null }`.

**Analytics Actions:**

5. `getAnalyticsSummary()` — Calculate and return:
   - **This month's bookings count:** Count bookings with `created_at >= start of current month` and `status IN ('confirmed', 'car_on_the_way', 'car_delivered', 'completed')`.
   - **This month's revenue:** Sum of `total_due` for those bookings.
   - **Fleet utilization:** Count of distinct `vehicle_id` values from bookings where `start_date <= today AND end_date >= today AND status IN ('confirmed', 'car_on_the_way', 'car_delivered')`. Divide by total active vehicles (`is_active = true AND is_available = true`). Return as percentage.
   - **All-time totals:** Total bookings ever, total revenue ever.
   - Return type: `{ thisMonth: { bookings: number, revenue: number }, allTime: { bookings: number, revenue: number }, fleet: { total: number, utilized: number, utilizationRate: number } }`.
   - Use `.select('total_due, status')` with date filters — do NOT select `*` (analytics should be lightweight).

All actions call `verifyAdmin()` first. Use `createAdminClient()` for all queries (admin needs to see all data regardless of RLS).
